# 机制解释

> **任务类型**：机制解释
> **触发条件**：用户问"X 为什么""X 怎么算""X 的推导原理""X 为什么有效"

---

## 执行深度

主 agent 直接分析 → Final（简单机制）
或 Round 1(A) → Final（需定位机制在原文中的位置）
或 Round 1(A+B) → Final（需外部理论验证）

这是唯一可能不需要任何子 agent 的任务类型。关键在于判断机制解释是否需要外部证据。

## 判断是否需要子 agent

| 情况 | 处理方式 |
|------|---------|
| 机制全部在原文中（公式、推导、算法步骤） | 主 agent 直接分析，不派发子 agent |
| 机制在原文中但需要定位在长篇材料中的位置 | 派发 A 做结构定位 |
| 机制涉及原文未引用的外部理论 | 派发 A（定位）+ B（外部理论查证） |
| 机制跨多个学科或有争议 | 派发 A + B + C |

## 分析框架

机制解释的核心是**因果链拆解**：

1. **推导起点**：这个机制/公式/算法的物理或数学前提是什么？
2. **推导过程**：从前提如何一步步到达结论？中间有哪些关键假设？
3. **参数含义**：每个参数/变量在物理或语义上代表什么？
4. **数值验证**：用具体数值走一遍推导，验证逻辑自洽性
5. **适用边界**：在什么条件下这个机制失效或需要修正？

## Pre-final Gate

- 因果链已完整拆解（从前提到结论无断裂）
- 每个关键假设已标注（是定理、经验假设、还是近似）
- 如有数值验证，计算过程已展示
- 区分了"原文直接给出"和"主 agent 推断"的部分

## 输出重点

压缩为"现象—推导起点—因果链—参数含义—数值验证—适用边界"。用连续段落表述，公式推导用逐步展开的方式呈现。

## 典型编排

```
情况 1（简单机制）：
  主 agent 直接读原文 → 拆解因果链 → Final

情况 2（需定位）：
  Round 1 A：定位机制在原文中的位置、相关公式和实验
  Checkpoint：定位完成 → Final

情况 3（需外部验证）：
  Round 1 A：定位
  Round 1 B：外部理论查证
  Checkpoint：证据充足 → Final

Final:
  围绕机制输出"推导起点—因果链—参数—验证—边界"
```
