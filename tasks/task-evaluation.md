# 判定/评价

> **任务类型**：判定/评价
> **触发条件**：用户问"X 是否成立""实验是否支撑 claim""这个方法好不好""ablation 够不够"

---

## 执行深度

Round 1(A+B) → Final（简单判定）
或 Round 1(A+B+C) → Verification → Final（需要外部证据支撑的判定）

## 子 agent 策略

- **必须**：
  - A（证据提取）—— 从原文中提取支持/反对待判定命题的所有证据（实验数据、ablation、引用、理论推导）
  - B（外部对照）—— 查同领域其他工作对相同命题的判断、benchmark 结果、复现报告
- **可选**：
  - C（理论边界）—— 当判定涉及理论前提是否成立时

## 判定框架

1. **待判定命题**：明确写出需要判定的具体命题（不是模糊的"好不好"）
2. **判定标准**：用什么标准判定（实验显著性、理论一致性、benchmark 对比、ablation 完整性）
3. **支持证据**：原文中直接支持该命题的证据，标注证据强度（直接实验 > 间接引用 > 理论推导 > 作者声称）
4. **反对证据/缺口**：原文中缺失的证据、矛盾的数据、未覆盖的边界条件
5. **结论强度**：STRONG（多源直接支持）/ MODERATE（单源支持或间接支持）/ WEAK（仅作者声称或推断）

## Pre-final Gate

- 每个待判定命题都有独立的判定段落
- 结论强度已标注，且 WEAK 结论已明确降级表述
- 支持证据和反对证据/缺口都已列出

## 输出重点

压缩为"命题—判定标准—支持证据—反对证据/缺口—结论强度—边界"。不展开创新点分析，不要求跨领域映射表（除非判定本身涉及跨学科理论前提）。

## 典型编排

```
Round 1:
  A：原文证据提取（支持 + 反对）
  B：外部对照（同领域 benchmark、复现、批评）

Checkpoint:
  证据充足 → Final
  证据有缺口 → 补充 C（理论边界）或进入 Verification

Verification（如需要）:
  关键断言引用核查

Final:
  按命题输出判定结论 + 强度标注
```
