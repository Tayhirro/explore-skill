# 系统设计分析

> **任务类型**：系统设计分析
> **触发条件**：用户问"这个系统怎么设计的""架构是什么""模块之间怎么配合""为什么选这个方案"

---

## 执行深度

主 agent 直接分析 → Final（简单系统、文档齐全）
或 Round 1(A) → Final（需要通读长文档或代码库）
或 Round 1(A+B) → Final（需要对比替代方案或评估设计选择）

## 与"创新点分析"的区别

- 创新点分析关注"改了什么、为什么比旧方法好"——重点是**差异**
- 系统设计分析关注"怎么运作、为什么这样设计"——重点是**整体架构和数据流**

## 子 agent 策略

| 情况 | 处理方式 |
|------|---------|
| 系统文档完整、架构清晰 | 主 agent 直接分析 |
| 文档长或分散（多个 README、论文、代码） | 派发 A 做结构理解 |
| 需要评估设计选择的合理性 | 派发 A（结构）+ B（替代方案对比） |
| 涉及跨领域设计模式 | 派发 A + B + C |

## 分析框架

1. **整体架构**：系统的顶层组织方式，核心组件及其职责
2. **数据流**：输入如何经过各模块处理变为输出，关键中间状态
3. **设计决策**：每个关键设计选择的理由、替代方案、取舍
4. **模块接口**：模块之间如何通信、依赖关系、耦合程度
5. **扩展性与局限**：当前架构在哪里容易扩展、在哪里会遇到瓶颈

## Pre-final Gate

- 架构图或模块关系已描述（文字或图示）
- 数据流已从输入到输出完整追踪
- 关键设计决策已标注理由和替代方案
- 局限性已列出

## 输出重点

压缩为"整体架构—数据流—设计决策—接口—扩展性与局限"。按模块或数据流阶段组织，不按论文章节组织。不要求跨领域映射表（除非用户问"这个设计模式来自哪里"）。

## 典型编排

```
情况 1（简单系统）：
  主 agent 直接分析 → Final

情况 2（长文档）：
  Round 1 A：结构理解（架构、模块、数据流）
  Checkpoint → Final

情况 3（评估设计选择）：
  Round 1 A：结构理解
  Round 1 B：替代方案对比
  Checkpoint → Final

Final:
  系统设计分析笔记（按架构/数据流组织）
```
