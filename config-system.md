# config-system.md — 系统/代码类对象配置

> **适用对象**：软件系统、开源项目、代码库、架构设计、技术博客中的系统描述。
> **加载时机**：当 SKILL.md 步骤 0 检测到分析对象为系统或代码时，使用 Read 工具加载本文件。

---

## 一、任务类型分流表

分流后，使用 Read 工具加载对应的 `tasks/task-*.md` 获取该类型的详细执行协议（Round 结构、子 agent 策略、pre-final gate、输出要求）。

| 任务类型 | 触发条件 | 最低执行深度 | config |
|---------|---------|------------|--------|
| 单系统概览 | 用户问"这个系统/项目是什么" | Round 1(A) → Final | `tasks/task-overview.md` |
| 单系统深度分析 | 用户问"架构设计/模块原理/设计决策" | Round 1(A+B+C) → Round 2(逐模块/机制) → Final | `tasks/task-innovation.md` |
| 多系统对比分析 | 用户给出 ≥2 个系统并要求逐一分析或对比 | Round 1(A+B+C) → Round 2(逐系统逐机制) → Verification → Final | `tasks/task-multi-compare.md` |
| 机制解释 | 用户问"X 机制为什么/怎么实现/算法原理" | 主 agent 直接分析 → Final（简单）；或 Round 1(A) → Final（需定位） | `tasks/task-mechanism.md` |
| 判定/评价 | 用户问"设计是否合理""性能是否达标""方案适不适合" | Round 1(A+B) → Final（简单）；或 + Verification → Final（需外部证据） | `tasks/task-evaluation.md` |
| 领域全景 | 用户问"某类系统的生态/演进" | Round 1(仓库+综述+最新进展) → Final | `tasks/task-domain.md` |
| 混合型 | 多系统 + 设计决策 + 领域分析 | 取各类型最高要求的并集 | 加载所有涉及的 task-*.md |

分流结果必须记录在 research_state.md 顶部，后续每轮 checkpoint 时回查。
主 agent 不得在 checkpoint 时降级执行深度，除非用户明确同意。

## 二、Round 1 角色映射

| 角色 | 职责范围 |
|------|---------|
| A（结构理解） | 代码结构、模块关系、数据流、API 设计、核心抽象、关键设计决策 |
| B（对比与定位） | 同类系统对比、替代方案、技术选型理由、设计权衡（trade-off）分析 |
| C（理论与跨学科依据） | 设计模式、工程原则、分布式理论、性能理论、可靠性理论、架构反模式 |

- A 必须存在。B/C 按任务类型要求（参见分流表的最低执行深度）。
- 如果对象数量 ≥4，可以增设 D/E 等辅助角色按对象分组，但 A/B/C 三个证据维度不可省略。

## 三、跨领域概念映射——学科建议

根据系统设计决策的类型，推荐的映射方向：

| 设计决策类型 | 推荐映射领域（≥2） | 典型概念 |
|------------|-----------------|---------|
| 涉及模块化/解耦/分层 | 生物学 + 语言学 | 模块性/层次性、语法-语义分离、关注点分离 |
| 涉及一致性/可用性/分区 | 分布式理论 + 控制论 | CAP 定理、BASE 一致性、Paxos/Raft 共识、反馈控制 |
| 涉及可扩展性/演进 | 组织理论 + 城市规划 | Conway 定律、Brooks 法则、增量式城市规划 |
| 涉及容错/韧性/降级 | 可靠性工程 + 生态学 | FTA/FMEA/RCA、拜占庭容错、生态系统韧性/冗余 |
| 涉及接口/协议/契约 | 语言学 + 经济学 | 接口隔离原则、契约式设计、协议兼容性、交易成本 |
| 涉及缓存/预取/调度 | 认知科学 + 运筹学 | 工作记忆模型、预取策略、排队论、调度算法 |
| 涉及安全/隔离/权限 | 安全工程 + 政治学 | 最小权限原则、零信任架构、权力制衡、信息流控制 |

如果某个设计决策确实无法建立跨领域映射（如纯配置选择），必须写明"无合适映射"并给出理由。

## 四、优先搜索来源

1. **官方文档和技术报告**：
   - 系统官方文档、白皮书、技术博客（如 Netflix Tech Blog、Uber Engineering、Google Research Blog）。
   - GitHub 仓库 README、ADR (Architecture Decision Records)、RFC 文档。

2. **架构设计文档和书籍**：
   - Martin Fowler's Blog、InfoQ、HighScalability。
   - 权威书籍：《Designing Data-Intensive Applications》《Clean Architecture》《Patterns of Enterprise Application Architecture》《System Design Interview》。

3. **顶会论文（如涉及研究系统）**：
   - SOSP、OSDI、EuroSys、NSDI、SIGCOMM、VLDB、SIGMOD、ATC、ASPLOS。
   - 工业系统会议：CIDR、SOCC。

4. **工程最佳实践和社区讨论**：
   - Stack Overflow、Hacker News、Reddit (r/programming, r/system-design)。
   - 技术博客和案例研究（case study）。
   - 性能基准：TechEmpower、YCSB、JMH、SPEC。

5. **浏览器接管检索**：
   - 在线文档、GitHub 仓库、性能基准站点等需要交互式浏览的场景。
   - 浏览器检索结果必须记录查询词、来源 URL、检索日期。

6. **证据使用规则**：
   - 优先引用官方文档、正式论文和权威技术书籍。
   - 技术博客可作为工程实践证据但需标注来源类型。
   - 社区讨论（HN/Reddit）只能作为定位线索，不能作为最终证据。
   - 区分"官方设计意图"和"社区使用经验"。

## 五、补充 Few-shot 示例

**例 S1**：用户问"这个系统的架构能迁移到我的项目吗？"
- 如果项目背景已经清楚，Round 1 可按"原系统理解 + 迁移判断"拆：
  - A：原系统结构、模块、数据流、约束。
  - B：迁移条件、风险、维护成本和失败边界。
- 如果用户项目背景不清楚，先问缺失约束或由主 agent 列假设，再决定是否需要外部对照。

**例 S2**：用户问"这两个系统的架构有什么不同？"
- Round 1：
  - A：两个系统的结构理解（可各派一个子 agent 或合并）。
  - B：设计哲学和取舍的对比（如一致性 vs 可用性、单体 vs 微服务）。
  - C：性能/可扩展性/可靠性的理论分析。
- Checkpoint：抽取关键设计决策逐一分析。
- Round 2：每个关键设计决策单独分析因果链和边界。

**例 S3**：用户问"这个项目的代码结构为什么这样设计？"
- Round 1：
  - A：代码结构理解，提取模块边界、依赖关系、核心抽象。
  - B：对比同类项目的不同设计选择，分析为什么选择当前方案。
  - C：设计模式/架构模式的理论依据（如 MVC、事件驱动、Actor 模型）。
- Final：输出"设计决策—动机—机制—权衡—边界"。

## 六、Section 4 补充 Few-shot（子 agent 查证编排）

- **问题**：某个系统的架构是否存在单点故障？
  - A：查系统文档和源码中的故障处理路径；
  - B：查分布式系统容错理论（CAP、Paxos、拜占庭容错）；
  - C：查该系统的生产事故报告和 post-mortem。

- **问题**：微服务 vs 单体架构在特定场景下哪个更合适？
  - A：查架构决策文献和 case study；
  - B：查组织理论（Conway 定律）和团队规模数据；
  - C：查性能基准和运维成本对比。

- **问题**：某个系统的数据一致性保证机制是什么？
  - A：查系统源码和官方文档中的一致性协议实现；
  - B：查分布式一致性理论（线性一致性、顺序一致性、因果一致性）；
  - C：查该系统在极端场景下的一致性行为报告。
