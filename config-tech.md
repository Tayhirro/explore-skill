# config-tech.md — 技术方案/框架类对象配置

> **适用对象**：技术框架、协议规范、API 设计、技术方案、开发工具链。
> **加载时机**：当 SKILL.md 步骤 0 检测到分析对象为技术方案或框架时，使用 Read 工具加载本文件。

---

## 一、任务类型分流表

| 任务类型 | 触发条件 | 最低执行深度 |
|---------|---------|------------|
| 单框架概览 | 用户问"这个框架/方案是什么" | Round 1(A) → Final |
| 单框架深度分析 | 用户问"设计原理/适用场景/核心机制" | Round 1(A+B+C) → Round 2(逐机制/决策) → Final |
| 多框架对比分析 | 用户给出 ≥2 个框架并要求逐一分析或对比 | Round 1(A+B+C) → Round 2(逐框架逐机制) → Verification → Final |
| 领域全景 | 用户问"某类技术的生态/演进" | Round 1(综述+最新进展+标准) → Final |
| 混合型 | 多框架 + 设计原理 + 领域分析 | 取各类型最高要求的并集 |

分流结果必须记录在 research_state.md 顶部，后续每轮 checkpoint 时回查。
主 agent 不得在 checkpoint 时降级执行深度，除非用户明确同意。

## 二、Round 1 角色映射

| 角色 | 职责范围 |
|------|---------|
| A（结构理解） | 方案定位、核心问题、设计目标、组件关系、API/接口设计、配置体系 |
| B（对比与定位） | 替代方案对比、业界标准做法、该方案的差异化价值、迁移成本和学习曲线 |
| C（理论与跨学科依据） | 底层原理、理论基础、行业最佳实践的理论依据、标准化组织的规范 |

- A 必须存在。B/C 按任务类型要求（参见分流表的最低执行深度）。
- 如果对象数量 ≥4，可以增设 D/E 等辅助角色按对象分组，但 A/B/C 三个证据维度不可省略。

## 三、跨领域概念映射——学科建议

根据技术方案的设计决策类型，推荐的映射方向：

| 设计决策类型 | 推荐映射领域（≥2） | 典型概念 |
|------------|-----------------|---------|
| 涉及标准化/兼容性 | 标准规范 + 经济学 | RFC/IETF/ISO/IEEE 标准化流程、网络效应、锁定效应 |
| 涉及抽象/DSL/语法 | 语言设计 + 认知科学 | 表达力-可判定性权衡、认知负荷理论、渐进式类型化 |
| 涉及性能/效率/延迟 | 计算理论 + 物理 | 大O分析、Amdahl定律、热力学类比、信息论下界 |
| 涉及安全/隔离/信任 | 安全工程 + 博弈论 | 威胁模型、信任边界、机制设计、激励兼容性 |
| 涉及可扩展/插件化 | 架构设计 + 生态学 | 开放/封闭原则、依赖倒置、生态位分化、共生关系 |
| 涉及状态管理/一致性 | 分布式理论 + 数据库理论 | ACID/BASE、CRDT、事件溯源、CQRS |
| 涉及人机交互/DX | HCI + 心理学 | 菲茨定律、希克定律、心流理论、认知负荷 |

如果某个设计决策确实无法建立跨领域映射（如纯配置选择），必须写明"无合适映射"并给出理由。

## 四、优先搜索来源

1. **官方规范和标准文档**：
   - RFC（IETF）、W3C Recommendations、ISO/IEEE 标准。
   - 框架官方文档（如 React docs、Kubernetes docs、Rust RFC）。

2. **参考实现和白皮书**：
   - GitHub 参考实现、设计文档、技术白皮书。
   - 提案文档（如 TC39 proposals、PEP、Rust RFC）。

3. **技术博客和案例研究**：
   - 框架作者/维护者的博客（如 Dan Abramov、Linus Torvalds）。
   - InfoQ、The Morning Paper、ACM Queue。
   - 企业技术博客（Google、Netflix、Cloudflare、Vercel）。

4. **社区讨论和采纳度指标**：
   - Stack Overflow 趋势、GitHub Stars/Issues/Discussions。
   - State of JS / State of CSS / JetBrains Developer Survey。
   - Hacker News 讨论、Reddit (r/programming, 对应技术子版)。

5. **浏览器接管检索**：
   - 在线规范文档、兼容性矩阵（caniuse）、官方 changelog 等需要交互式浏览的场景。
   - 浏览器检索结果必须记录查询词、来源 URL、检索日期。

6. **证据使用规则**：
   - 优先引用官方规范文档和参考实现。
   - 框架作者的博客/演讲可作为设计意图证据。
   - 社区讨论和 Survey 可作为采纳度和满意度证据，但不能作为技术正确性证据。
   - 区分"规范定义的行为"和"实际实现的行为"。

## 五、补充 Few-shot 示例

**例 T1**：用户问"这个框架比替代方案好在哪里？"
- Round 1：
  - A：框架的结构理解（核心抽象、API 设计、配置体系）。
  - B：与 2-3 个主要替代方案的对比（设计理念差异、适用场景差异）。
  - C：底层原理分析（为什么在特定场景下性能/开发效率更好）。
- Checkpoint：抽取关键设计决策逐一分析。
- Final：输出"设计决策—对比—底层原理—适用边界"。

**例 T2**：用户问"这个技术方案能解决我的问题吗？"
- Round 1：
  - A：方案的核心机制和适用条件。
  - B：该方案在类似场景中的实际案例和失败案例。
- Checkpoint：判断用户场景是否落在方案的适用范围内。
- Final：输出"方案机制—适用条件—用户场景匹配度—风险"。

**例 T3**：用户问"这个领域有哪些主流框架，它们的区别是什么？"
- Round 1：
  - A：领域全景，梳理主要框架和演进脉络。
  - B：各框架的设计哲学和技术路线对比。
  - C：标准化进展和行业采纳情况。
- Final：输出"领域地图—框架对比—适用场景—演进趋势"。

## 六、Section 4 补充 Few-shot（子 agent 查证编排）

- **问题**：某个框架的性能声明是否成立？
  - A：查官方 benchmark 和测试方法论；
  - B：查独立第三方 benchmark 和社区复现；
  - C：查计算理论中的下界分析（是否存在理论不可能更好的情况）。

- **问题**：某个技术方案的生产环境可靠性如何？
  - A：查官方 case study 和客户证言；
  - B：查社区报告的生产事故和问题；
  - C：查可靠性工程理论对该方案的适用性分析。

- **问题**：某个框架的 API 设计是否符合人机交互最佳实践？
  - A：查框架文档中的 API 设计原则和示例；
  - B：查 HCI 和认知科学中关于 API 可用性的理论；
  - C：查开发者 Survey 和社区反馈中的痛点。
