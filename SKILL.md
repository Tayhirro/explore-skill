# Skill：深度研究/分析

## 目标

研究一篇论文、一个 topic、一个系统设计或一个创新点时，让 AI 帮我完成：

1. 表层理解；
2. 第一性原理分析；
3. 创新点拆解；
4. 因果机制分析；
5. 独立子问题调研；
6. 学术依据查证；
7. 笔记化压缩；
8. 迁移到自己的研究方向和系统设计。

核心思路：

> 主 agent 负责建立结构、抽象问题、设计子问题；
> 子 agent 负责清空上下文、独立调研、查证学术依据；
> 子 agent 只上报后续候选问题，不创建子子 agent；
> 主 agent 再负责 triage、继续调度、综合、压缩、改写成自己的研究笔记。

### 对象配置文件加载

本 skill 的核心方法论适用于所有分析对象类型。对象特定的分流表、角色映射、跨领域映射建议和搜索来源保存在独立的配置文件中：

| 配置文件 | 适用对象 |
|---------|---------|
| `config-paper.md` | 学术论文、算法、研究综述 |
| `config-system.md` | 软件系统、开源项目、代码库、架构设计 |
| `config-tech.md` | 技术框架、协议规范、API 设计、技术方案 |

**加载规则**：在步骤 0 判断对象类型后，使用 Read 工具读取对应的 config 文件，获取分流表、角色映射和跨领域映射要求。如果分析对象跨多个类型（如论文中描述的开源系统），读取最相关的 config 文件，并在规划摘要中说明。

## 1. 先做表层理解

收到用户给出的论文、topic、系统名、创新点或研究材料后，先直接解释基本内容，避免一开始过度抽象。

不要把本节写成"推荐提问"返回给用户。用户已经提出研究请求时，AI 应直接按下面维度完成表层理解；如果材料不足，先基于已有信息分析，并明确标出还需要补充什么。

执行维度：

```md
先解释【论文 / topic / 系统名】大概在做什么，不要直接上升到哲学层。

必须回答：

1. 它研究的问题是什么？
2. 它输入什么、输出什么？
3. 它的核心流程是什么？
4. 它和已有方法相比有什么明显不同？
5. 它最重要的模块有哪些？
6. 每个模块在流程中承担什么作用？
```

目的：

> 先建立直观理解，知道系统在做什么、怎么做、模块之间怎么连接。

---

## 2. 做通用深层分析

完成表层理解后，继续分析底层结构。

AI 应直接使用下面的框架分析当前对象。

执行维度：

```md
从通用深层结构分析这个对象，不要停留在模块描述或表面总结。

按下面维度选择性分析；不是每个任务都要填满所有维度。

1. 对象与边界：
   - 当前真正要研究 / 解释 / 判定 / 映射的对象是什么？
   - 它不研究什么？哪些内容只是背景或辅助材料？

2. 核心问题：
   - 它要回答什么问题？
   - 这个问题属于内容理解、机制解释、判定评价、概念映射、方法创新，还是混合型？

3. 依据与可信度：
   - 结论凭什么成立？
   - 依据来自原文、实验、对照测试、性能评估、理论证明、综述、经典理论、外部工具、复现结果，还是综合推断？
   - 哪些是文献直接支持，哪些只是合理类比或主 agent 的综合判断？

4. 路径 / 机制 / 组织方式：
   - 如果是机制解释：因果链是什么？
   - 如果是综述或文章理解：它如何组织领域、章节、taxonomy、观点和证据？
   - 如果是系统分析：模块之间如何连接，信息如何流动？
   - 如果是判定问题：判定标准如何作用到证据上？
   - 如果是概念映射：源概念如何映射到目标理论？

5. 形式化视角：
   - 是否可以形式化为 MDP、POMDP、Bayesian inference、tree search、bandit、optimization、credit assignment、information gain、constraint satisfaction 等？
   - 如果形式化会造成误导，就明确说明不强行形式化。

6. 适用范围与失败边界：
   - 结论在什么条件下成立？
   - 什么反例、数据分布、任务设置、评估标准或理论前提会让它失效？

7. 后续价值（可选）：
   - 只有当任务需要时，才分析它对理解论文、构造系统、设计实验、迁移应用或发现研究机会有什么启发。

8. 第一性原理分析（当任务要求"本质""原理""为什么有效"时执行）：
   按以下步骤操作，不要跳过任何步骤：
   a. 分解：把当前对象拆解到不可再分的基本假设、约束和公理。问"这个设计/机制/方法依赖哪些最基本的前提？如果去掉某个前提，它还成立吗？"
   b. 穷举替代：对每个基本前提，问"在同样的约束下，还有哪些本质不同的方案可以达到同样的目标？"如果只能想到当前方案，说明分解还不够深。
   c. 重建：从基本前提重新推导，当前方案是必然最优解、还是众多可行解中偶然被选中的一个？选择它的原因是什么（计算效率、数据可得性、历史路径依赖、还是理论下界）？
   d. 对比：把推导出的"必然性"和作者/设计者声称的动机对比。如果一致，说明设计有坚实的理论基础；如果不一致，说明实际驱动力可能是工程便利、数据偏好或领域惯例，需要标注为"综合推断"。
   输出格式：每个核心机制/决策用一段连续表述覆盖 a-d，不要用字段清单。先写基本前提，再写替代方案空间，最后写必然性判断。
```

目的：

> 从"它在说什么"进入"它研究什么、凭什么成立、如何组织推理、在哪里有效、还能迁移到哪里"。

## 3. 从用户问题推进到"问题—调研—机制映射"

先由主 agent 判断当前问题更适合什么分析框架，再决定使用什么字段。不要把所有问题都强行归入固定类型，也不要把所有问题都写成"旧方法缺陷 / 本质缺陷"。下面类型只是常见参考；如果用户的问题是混合型，就组合使用相应字段。只有在问题和证据清楚之后，才把它映射到对象中的具体机制或模块。

执行流程：

**注意：步骤 0-7 是主 agent 的内部规划过程，不向用户输出分析结果。** 主 agent 快速扫描材料、判断类型、设计分析框架和拆分方案，然后将规划摘要呈现给用户确认（步骤 8）。实质分析的执行和向用户输出结果，在规划确认门通过后才开始。Section 1（表层理解）和 Section 2（通用深层分析）的分析维度在此阶段用于指导规划，完整输出在确认后按规划执行。

```md
0. 任务类型分流（强制，在分析框架判断之前执行）：
   在进入分析框架之前，先判断任务类型并锁定最低执行深度。
   
   **对象类型检测与配置加载**：
   - 判断分析对象属于论文/算法、系统/代码、还是技术方案/框架。
   - 使用 Read 工具读取对应的 config-*.md，获取该对象类型的分流表和角色映射。
   - 将分流表中的任务类型和最低执行深度应用到当前任务。
   
   "对象"可以是论文、系统、技术方案、开源项目、算法、框架等任何被分析目标。
   分流结果必须记录在 research_state.md 顶部，后续每轮 checkpoint 时回查。
   主 agent 不得在 checkpoint 时降级执行深度（如从"多对象对比分析"降为"领域全景"），除非用户明确同意。

1. 判断分析框架：
   - 由主 agent 根据用户问题自行判断，不要求输出固定分类名。
   - 常见参考：
      - 创新点 / 方法改进 / 系统对比：例如"这篇文章的创新点是什么""它改了哪里""解决了什么痛点""有什么值得学习和迁移"，使用"旧方法缺陷、本质缺陷、设计哲学、修改点、机制与迁移价值"。
     - 内容 / 结构解释：例如"这篇文章在干什么""这篇综述讲了什么""这个系统由哪些模块组成"，使用"研究对象、文章结构、核心问题、模块关系、主线论证"。
       - 这类任务的主对象是原文内容本身，不是相关背景知识。必须先解释文章讲了什么、怎么组织、作者想回答什么问题，再决定是否需要外部调研。
     - 机制解释：例如"为什么 X 会发生""X 为什么有效"，使用"待解释现象、因果机制、理论依据"。
     - 判定 / 评价：例如"X 是否是 Y""这个创新点重要吗""实验是否支撑 claim"，使用"待判定命题、判定标准、支持证据、反例边界、结论强度"。
     - 概念映射：例如"X 能否形式化为 POMDP / VOI / belief update"，使用"源概念、目标理论、映射关系、映射限制"。

2. 选择通用分析字段：
   - 按当前问题选择必要字段，不需要保留无关字段。
   - 对象与边界：当前真正研究 / 解释 / 判定 / 映射的对象是什么？哪些内容不在范围内？
   - 核心问题：当前要回答的是内容理解、机制解释、判定评价、概念映射、方法创新，还是混合问题？
   - 依据与可信度：判定标准、解释框架、证据来源是什么？哪些是直接支持，哪些是综合推断？
   - 路径 / 机制 / 组织方式：文章如何组织论证，系统如何流动信息，机制如何产生效果，概念如何映射？
   - 适用范围与失败边界：结论在什么条件下成立，什么反例或前提变化会让它失效？

3. 拆成需要查证的问题：
   - 拆分不是强制动作。只有当问题本身包含多个独立子命题、多个章节 / 模块、多个证据链，或需要外部查证时，才拆成多个问题。
   - 每个问题必须来自明确缺陷、文章结构 / 模块关系、待解释现象、待判定命题或概念映射关系，不是孤立模块名。
   - 不要在缺少依据时先写确定性结论。

4. 判断每个问题是否需要子 agent 调研：
   - 是否开启子 agent 不取决于问题类型，而取决于是否需要拆分材料或外部证据。
   - 对原文内容 / 结构解释：如果文章很短、结构简单，主 agent 直接解释即可，不要为了使用多 agent 而强行派发子 agent；如果是长论文、综述、多模块系统或跨学科材料，需要启动子 agent 时，必须保留一个贯穿全文的 A 子 agent 负责整体概括，覆盖对象主线、章节或模块结构 / taxonomy、核心断言和整体结构。其余 B/C/D 子 agent 按需启动，可以按章节、模块、观点簇、学科背景、证据链或外部 retrieval 拆分。最终由主 agent 汇总各子 agent 的结构化结果，形成对整体对象的概括和必要补充。
   - 对机制解释、判定 / 评价、概念映射类问题：如果涉及深层机制、学术依据、跨学科理论、外部论文支持、争议判断或反例边界，先按第 4、5 节开启独立子 agent。
   - 子 agent 只查证自己负责的章节、模块、理论依据、证据、适用范围、边界、反例和不确定部分。
   - 子 agent 不能自行创建子子 agent；如果发现需要继续深入的问题，只能写成 follow-up candidate 返回给主 agent。

5. 综合调研结果：
   - 区分"文献直接支持"和"基于文献的综合推断"。
   - 去掉泛泛解释，只保留能支撑内容理解、判定、机制解释或概念映射的结论。

6. 建立分析映射：
   - 对象中的哪个模块、算法、流程、设计决策或实验设置回应了这个问题？
   - 如果是核心单元类问题：该机制如何体现第 2 节的设计哲学？
   - 如果是内容 / 结构解释：该模块 / 章节在整体结构中承担什么作用？
   - 如果是机制解释、判定 / 评价、概念映射类问题：该机制如何支撑解释、判定、评价或理论映射？
   - 如果是机制问题，再说明该机制如何改变系统行为。
   - 结论适用于什么范围？有哪些边界、反例或不确定部分？

7. 处理后续候选问题：
   - 子 agent 返回的 follow-up candidate 不自动等于下一轮任务。
   - 主 agent 必须先 triage：合并重复候选、判断影响范围、决定是否提升为下一轮调研任务。
   - 被提升的问题进入新的 round 或新的问题研究目录；未提升的问题标记为 deferred / dropped，不继续展开。

8. 规划确认门（强制，无条件执行）：
   在完成步骤 0-7 的分析规划后、开始任何实质分析或执行之前，主 agent 必须向用户输出一份规划摘要，并使用 AskUserQuestion 工具等待用户确认。无论任务类型、材料长度、是否需要子 agent，此门都不可跳过。
   
   规划摘要格式：
   ```
   ## 调研规划确认
   
   **对象类型**：【论文/算法 | 系统/代码 | 技术方案/框架】
   **任务类型**：【从 config 分流表填入，如"混合型：多论文对比分析 + 创新点本质"】
   **最低执行深度**：【如"Round 1(A+B+C) → Round 2(逐创新点) → Verification → Final"】
   **预估子 agent 数量**：【Round 1: X 个, Round 2: ~Y 个, Verification: 1 个】
   
   **Round 1 拆分方案**：
   - A：【结构理解 — 具体范围（参见 config 中的 A 角色职责）】
   - B：【对比与定位 — 具体范围（参见 config 中的 B 角色职责）】
   - C：【理论与跨学科依据 — 具体覆盖哪些学科（参见 config 中的 C 角色职责）】
   - D（如有）：【辅助维度 — 具体范围】
   
   **Round 2 拆分方案（预览）**：
   - 按核心单元逐一分析：U1 → 独立分析段, U2 → 独立分析段, ...
   - 每个核心单元覆盖：痛点→修改点→机制→证据→边界
   - （如任务类型不含 Round 2，写"不适用"并说明理由）
   
   **Verification 方案（预览）**：
   - 待查断言数量和来源：从 Round 1+2 结果中抽取 ≥5 条关键断言（如任务类型要求）
   - 核查维度：引用一致性、实验支持、claim-证据匹配
   - （如任务类型不含 Verification，写"不适用"并说明理由）
   
   **Final 输出预期**：
   - 输出结构和覆盖范围简述
   
   **已抽取的核心单元（预览）**：
   - U1：【对象X 的核心单元Y — 初步判断的本质】
   - U2：...
   
   **预估 token 消耗等级**：低 / 中 / 高 / 极高
   ```
   
   输出规划摘要后，使用 AskUserQuestion 工具询问用户：
   - "以上调研规划是否符合你的预期？"
   - 选项：(1) 确认执行 (2) 调整深度（更浅/更深） (3) 调整拆分方案 (4) 我来补充说明
   
   用户确认后，将确认后的规划摘要完整写入 `tmp/research_state.md` 的"已确认规划"小节，作为后续 checkpoint 回查的基准。
   用户要求调整时，主 agent 修改规划摘要中的对应项，再次确认后执行。
   此门没有例外。即使任务类型为"单对象概览"、材料很短、或不需要子 agent，也必须输出规划摘要并获得用户确认。
```

本节输出不是最终笔记，而是主 agent 和子 agent 共享的结构化中间结果。具体可以理解成一组临时字段，例如 `design_philosophy_summary`、`causal_issue_cards`、`subagent_research_tasks`、`evidence_mapping`，用于决定下一步 action、是否派发子 agent、以及最后如何压缩成笔记。

### 3.1 主 agent 调度的迭代深化

本 skill 中的"递归"不是子 agent 继续创建子子 agent，而是主 agent 统一调度的迭代深化。子 agent 只能完成当前分配任务，并把新发现的问题作为 `followup_candidates` 返回；只有主 agent 可以判断是否进入下一轮调研、是否创建新的问题研究目录、是否更新已有 `research.md`。

执行原则：

```md
1. 第一轮先建立骨架：
   - 如果输入是论文、综述或文章，必须先完成原文结构理解。
   - 原文或对象结构理解覆盖对象主线、章节或模块结构 / taxonomy、核心断言、关键模块、声称的贡献或设计目标和证据组织方式。

2. 子 agent 只上报候选问题：
   - 可以指出当前任务无法回答的问题、证据缺口、替代解释、跨领域依据需求、失败边界和可迁移启发。
   - 不可以自行创建子子 agent。
   - 不可以擅自扩大任务范围继续调研。
   - 不可以把候选问题写成已验证结论。

3. 主 agent 每轮都做 triage：
   - 收集所有 `followup_candidates`；
   - 合并重复或高度相似的问题；
   - 判断每个候选问题是否影响最终对内容、核心单元、机制、证据强度或迁移价值的判断；
   - 只把 high-impact 问题提升为下一轮调研任务；
   - 为被提升的问题创建或更新 `tmp/<编号>-<问题短名>问题研究/research.md`；
   - 记录 `parent_issue`、`derived_from`、`source_subagent_output` 和 promotion reason。

4. Round 1 后必须做 checkpoint：
   - 回查 `tmp/research_state.md` 的"已确认规划"小节，确认 Round 1 实际执行与用户确认的方案一致（子 agent 数量、角色、范围是否匹配确认的拆分方案）。如有偏离，记录偏离原因；如超出确认范围，需要用户明确同意。
   - 如果已经派发过子 agent，不允许直接写最终笔记。
   - 必须先更新 `research.md` 中已完成子任务的状态。
   - 必须收集所有 `followup_candidates` 到 `tmp/followup_candidates.md`。
   - 必须抽取与用户问题直接相关的 candidate units，例如核心创新点或设计决策、关键断言、待解释机制、实验缺口、可迁移原则。
   - 抽取时必须按以下标准判定和分级：
     
     **什么算一个独立的核心单元（candidate unit）：**
     - 它必须是一个可独立分析的决策或机制，而不是一个模块名或技术名词。"使用了 Transformer"不是候选单元；"用因果掩码 Transformer 替代 RNN 做时序预测，以解决长程依赖的梯度消失"才是。
     - 它必须改变了对象相对前序工作的行为或能力。如果去掉这个改动，对象的行为退化为已有方法或性能明显下降，则它是一个有效候选单元。
     - 对象类型适配：论文中对应"创新点"，系统中对应"关键设计决策"，技术方案中对应"核心机制选择"。

     **分级规则（每个候选单元必须标记一个级别）：**
     | 级别 | 判定标准 | Round 2 处理 |
     |------|---------|-------------|
     | **核心（Core）** | 改变了方法的根本范式或引入了全新的问题表述；去掉后对象失去其核心贡献 | 默认 high-impact，必须进入 Round 2 逐一分析 |
     | **次要（Secondary）** | 在已有范式内的改进（如效率提升、精度优化、新数据源适配）；去掉后仍保留核心贡献但性能下降 | 按用户要求决定是否进入 Round 2 |
     | **工程细节（Engineering）** | 实现层面的选择（如超参数、框架选型、训练技巧）；不影响方法的本质 | 不进入 Round 2，可在最终笔记中简要提及 |

     **数量指引：**
     - 每篇论文/每个对象通常有 1-3 个核心创新点和 2-5 个次要改进。如果抽取超过 5 个核心，很可能把次要改进误判为核心，需要重新分级。
     - 如果对象是综述或领域分析，candidate unit 不是创新点而是"关键断言"或"分类决策"，分级标准改为：核心断言（改变领域认知）、次要断言（补充证据）、背景信息（已知事实的复述）。
   - 每个 candidate unit 都必须标记 promoted / deferred / dropped / resolved，并写明理由。
   - 只有完成 checkpoint，才能进入下一轮调研或最终输出。

4.5. **轮次转换门（强制，不可跳过）**：
   在进入下一轮（Round N+1）之前，主 agent 必须完成以下全部步骤，**否则禁止派发任何下一轮的子 agent**：

   a. **更新 research.md**：将本轮所有子 agent 的状态更新为 completed / failed / partial，并记录每个子 agent 输出文件的路径。
   b. **收集 follow-up candidates**：将所有子 agent 返回的 follow-up candidates 合并到 `tmp/followup_candidates.md`，并完成 triage（合并重复、标记优先级、决定 promote / defer / drop）。
   c. **抽取/更新核心单元**：根据本轮结果抽取或修正 candidate units，按分级规则标记 Core / Secondary / Engineering，并标记 promoted / deferred / dropped / resolved。
   d. **写入下一轮子任务清单**：在 research.md 中写入 Round N+1 的完整子任务清单，每个子任务包含：编号、角色、方向、状态（pending）、输出文件路径。格式与本节"子任务清单"一致。
   e. **更新 research_state.md 执行清单**：在顶层执行清单中打勾当前轮次已完成项，并追加下一轮的待办项。

   **硬性约束：Round N+1 的任何一个子 agent 被派发时，research.md 中必须已存在该子任务的条目（编号、角色、方向、输出路径）。不允许"边跑边规划"。**

5. 候选问题升级条件：
   - 当前层无法仅凭原文直接回答；
   - 需要外部论文、综述、经典理论或最新进展查证；
   - 需要核对实验、对照测试、性能评估、引用或断言-证据一致性；
   - 涉及多个可能解释，需要比较支持强度；
   - 会明显影响对核心单元、痛点、贡献、失败边界或迁移价值的判断。

6. 默认 high-impact 的情况：
   - 用户要求"每个创新点 / 每个机制 / 每个 claim / 每个模块"分别分析时，每个被抽取出的核心单元默认 high-impact。
   - 用户只问文章某一个点时，该点本身默认 high-impact；背景、相关理论和对比方法只是第一轮证据扫描，不能替代对该点的单独分析。
   - 如果主 agent 决定不提升某个核心单元，必须写明为什么当前证据已经足够、继续调研不会改变结论。

7. 停止条件：
   - 没有新的 high-impact follow-up candidate；
   - 新候选只补充背景知识，不影响最终判断；
   - 已经能区分文献直接支持、综合推断和不确定部分；
   - 继续调研只会扩大范围，不能提高最终答案质量；
   - 用户设定的轮次、时间或深度预算已经达到。
```

派生问题的文件结构默认保持扁平：所有问题目录都放在同一个 `tmp/` 下，不强制创建多层嵌套目录。用 `research.md` 中的 `parent_issue`、`derived_from` 和 `source_subagent_output` 记录依赖关系，避免路径和上下文管理复杂化。

Round 1 拆分约束（强制）：

```md
1. Round 1 必须按"证据维度"拆分子 agent，禁止用"对象分组"（如把 3 篇论文/3 个系统交给一个子 agent）替代证据维度拆分。
2. 子 agent 角色按对象类型适配：读取对应的 config-*.md 获取 A/B/C 角色的具体职责定义。A（结构理解）必须存在，B（对比与定位）和 C（理论与跨学科依据）按任务类型要求。
3. 如果对象数量 ≥4，可以增设 D/E 等辅助角色按对象分组，但 A/B/C 三个证据维度不可省略。
4. Round 1 拆分方式必须与步骤 0 确定的最低执行深度一致，不允许用更粗的拆分替代要求的细粒度拆分。
```

执行清单（强制）：

```md
主 agent 必须在 `tmp/research_state.md` 顶部维护执行清单（紧跟"已确认规划"小节之后），每完成一项打勾，checkpoint 时逐项核对。每个问题研究目录下的 `research.md` 只负责该问题的子任务状态跟踪，不重复维护顶层执行清单。

执行清单根据步骤 0 的任务类型自动生成，格式如下：

## 执行清单

任务类型：【从分流表填入】
最低深度：【从分流表填入】

- [ ] 规划确认门已通过（用户已确认；已确认规划已写入 `tmp/research_state.md`）
- [ ] Round 1 按证据维度拆分（非按对象分组）
- [ ] Round 1 包含 A-结构理解
- [ ] Round 1 包含 B-对比与定位（如任务类型要求）
- [ ] Round 1 包含 C-理论与跨领域依据（如任务类型要求）
- [ ] 每个核心单元/机制已抽取并标记 high-impact（如任务类型要求）
- [ ] 轮次转换门已通过（research.md 已更新 Round 1 状态 + Round 2 子任务清单已写入）（如有 Round 2）
- [ ] Round 2 逐核心单元/机制分析完成（如任务类型要求）
- [ ] 跨领域概念映射表已填充（≥2 领域，如任务类型要求）
- [ ] Verification 轮完成（≥5 条断言，如任务类型要求）
- [ ] pre-final gate 全部通过

Checkpoint 时必须逐项核对清单。任何未完成项不允许跳过，除非用户明确同意。
```

多轮调研原则与通用示例：

```md
派发编队原则：
- 先看用户真正要回答的焦点，再看每个焦点需要哪些独立证据链；每个子 agent 应该承担一个清楚的判断维度，而不是承担一个形式位置。
- 如果材料短、问题浅、主 agent 能直接读完并给出可靠答案，就直接完成；如果材料长、证据链分散或问题要求外部对照，再进入 Round 1。
- 论文 / 综述 / 文章类任务需要派发时，优先建立"原文结构理解"；相关方法、实验核查、理论边界、迁移风险等方向按问题需要加入。
- Checkpoint 用 answer units 和 evidence ledger 判断下一轮，不用背景调研替代目标问题本身的分析。
- 更多对象特定的 few-shot 示例见各 config-*.md。

通用示例 1（概览型）：
- 用户问"X 在干什么？"
- Round 1：派发 A 做结构理解。
- Checkpoint：如果只是 overview 且 A 已覆盖主线和贡献，resolved。
- Final：输出"对象—主线—结构—贡献"。

通用示例 2（创新点深度分析）：
- 用户问"每个创新点是什么、解决什么痛点、机制和证据如何？"
- Round 1：按证据来源拆。
  - A：结构理解，提取主线、claim、贡献列表。
  - B：相关方法缺陷与 baseline 对比。
  - C：理论依据与外部证据。
- Checkpoint：抽取核心创新点 I1/I2/I3，每个默认 high-impact。
- Round 2：按创新点拆，每个创新点有独立的分析段落（痛点→修改点→机制→证据→边界）。
- Final：只有 Round 2 完成或明确 deferred 后才写最终笔记。

通用示例 3（机制解释型）：
- 用户问"X 机制为什么有效？"
- Round 1：A 定位 X 在整体结构中的位置和 claim；B 查 X 的相关理论；C 查原文证据是否直接支持 X。
- Checkpoint：X 本身默认 promoted。
- Round 2：D 单独研究 X 的因果链、替代解释、证据强度、失败边界。
- Final：围绕 X 输出"现象—机制—证据—边界"。
```

当任务需要派发子 agent、跨多轮继续、材料较长或需要避免上下文丢失时，主 agent 应在当前研究对象所在目录或本次笔记输出目录下新建 `tmp/` 文件夹，把这些字段保存为中间草稿文件，例如 `tmp/research_state.md`、`tmp/evidence_mapping.md`、`tmp/followup_candidates.md`。

每个需要子 agent 查证的因果问题，都必须在 `tmp/` 下单独建立一个问题研究目录，不能只把结果写进最终笔记。例如：

```text
tmp/
  research_state.md
  evidence_mapping.md
  followup_candidates.md
  01-单agent自我合理化问题研究/
    research.md
    rounds/
      round1-retrieval/
        A-文章主线与结构概括.md
        B-机器学习与LLM论文.md
        C-认知科学与心理学.md
        D-哲学与决策理论.md
      round2-gapfill/
        gap-01-失败边界补证.md
      round3-verification/
        verification-report.md
  02-工具调用value-of-information问题研究/
    research.md
```

目录名使用"编号 + 问题短名 + 问题研究"，例如 `tmp/01-单agent自我合理化问题研究/research.md`。每个问题目录只维护一个主 agent 工作文件 `research.md`，它只合并原本会拆开的 `task.md` 和 `subagent_tasks.md`：包括问题背景、任务边界、需要查证 / 不需要查证的范围、round1 / round2 / round3 的子任务清单和状态。不要再额外生成问题级 `task.md`、`subagent_tasks.md`。

如果某个问题由上一轮子 agent 的 `followup_candidates` 派生而来，在该问题的 `research.md` 顶部记录：

```md
parent_issue:
derived_from:
source_subagent_output:
promotion_reason:
status:
```

子 agent 只读取 `research.md` 中分配给自己的任务小节，并把调研结果写到对应轮次目录，例如 `rounds/round1-retrieval/A-机器学习与LLM论文.md`。各轮结果不要写进 `research.md`；`research.md` 只负责记录任务拆分和状态。主 agent 最终交付时，要读取 `research.md` 和 `rounds/` 下的结果文件，把有价值的部分合并进第 7 节的最终笔记格式，并清楚标明哪些内容来自文献直接支持、哪些是综合推断。

只有在用户需要看到阶段性结果时，才把它整理成"阶段性分析摘要"输出。不要额外生成"问题卡 / 因果问题卡 / 问题研究卡"；这些结构容易和子 agent 提示词模板重复。

如果需要派发子 agent，主 agent 只需要在对应问题目录下维护一个 `research.md`，但必须在这个文件里写清楚 sub tasks：

1. 在 `research.md` 的顶部维护该问题的子任务清单和状态（不是顶层执行清单，顶层清单在 `tmp/research_state.md`），根据问题需要自动生成并在每轮 checkpoint 时更新。
2. 在 `research.md` 的"任务边界"小节写总体背景、主问题、需要查证和不需要查证的范围。
3. 在 `research.md` 的"Subtasks / 子任务清单"小节列出每一轮、每个子任务、负责方向、状态和输出文件路径。
4. 在 `research.md` 的"Round 1：Retrieval 子任务"小节列出每个 retrieval 子 agent 的任务和范围。
5. 如果输入是论文、综述或文章，Round 1 的第一个子任务必须是"原文结构理解"，用于概括文章主线、章节 / taxonomy 和作者断言。如果输入是系统或技术方案，Round 1 的第一个子任务是"对象结构理解"，用于概括架构、模块关系和核心设计目标。
6. 如果进入 gap-fill、verification 或下一轮迭代深化，继续在同一个 `research.md` 里追加"Round 2：Gap-Fill 子任务""Round 3：Verification 子任务"或"Round N：Follow-up 子任务"清单。
7. 在 `research.md` 的"Follow-up Candidates / 后续候选问题状态"小节记录每个候选问题的状态：promoted / deferred / dropped / resolved。

`research.md` 是该问题的唯一主 agent 工作文件；它保存任务和 subtasks，不保存各轮调研结果。子 agent 不再新建独立任务文件或新的问题研究目录，只把结果写入 `rounds/<轮次>/<子任务名>.md`。

输出结构：

```md
## 分析框架与通用骨架摘要

- **分析框架：**
  - 主 agent 自行判断；常见参考包括：创新点 / 改进、内容 / 结构解释、机制解释、判定 / 评价、概念映射、混合型。

- **通用分析字段：**
  - 按当前问题选择必要字段；不要保留无关字段。
  - 常见字段：对象与边界；核心问题；依据与可信度；路径 / 机制 / 组织方式；形式化视角；适用范围与失败边界；后续价值（可选）。

- **核心判断：**
  - ...
```

子 agent 任务写法：

```md
## 子 agent 任务：<角色>-<方向短名>

- 轮次：round1-retrieval / round2-gapfill / round3-verification
- 角色：retrieval / gap-fill / verification
- 独立问题：
- 为什么需要单独查证：
- 需要查：
- 不需要查：
- 证据要求：
- 工作文件：`tmp/<问题研究目录>/research.md`
- 任务位置：`research.md` 中对应的 `Round X / <角色>-<方向短名>` 小节
- 输出位置：`tmp/<问题研究目录>/rounds/<轮次>/<角色>-<方向短名>.md`
```

目的：

> 把"创新点、解释问题、判定问题或概念映射问题"变成可查证的问题拆解，先查证关键依据，再落到机制映射。

---

## 4. 遇到关键因果问题时，单开或多开子 agent

当某个问题涉及深层机制、学术依据或跨学科概念时，单独抽成一个子问题，让独立 agent 清空上下文后调研。

适合单开子 agent 的情况：

```md
1. 上下文污染：当前解释可能受到上下文污染。
2. 学术依据：某个核心单元需要证明设计有效性，某个因果机制需要学术依据，需要引用论文、综述或经典理论。
3. 涉及跨学科内容：哲学、认知科学、计算神经科学、决策理论、POMDP、active inference 等跨学科内容。
4. 需要区分"文献直接支持"和"综合推断"。
```

如果同一个因果问题涉及多个求证方向，不要强行交给一个子 agent 完成。应采用"多轮子 agent 查证"：

```md
第 1 轮：Retrieval
- 目标：广泛覆盖不同领域和来源，先把可能相关的理论、论文、证据、反例找出来。
- 启动方式：主 agent 按问题、来源或学科创建若干同级 retrieval 任务；子 agent 只执行 `research.md` 中分配给自己的任务小节。
- 拆分决策：
  - 主 agent 不预设固定拆分方式，而是按"哪种拆分能产生相互独立、可互相校验的证据链"来决定。
  - 如果输入是论文、综述或文章，Round 1 的第一个 retrieval 子任务必须是"原文或对象结构理解"：先概括对象本身是什么、结构如何组织、核心断言是什么、哪些内容只是背景、哪些断言需要外部查证。
  - 原文或对象结构理解不做外部概念扩展，不把任务改写成"查某个相关领域是什么"；外部背景知识只能作为后续 retrieval 子任务或 gap。
  - 默认按证据链 / 问题方向拆，例如理论依据、机器学习论文、认知科学机制、失败边界。
  - 如果任务重点是找最新论文、确认 venue、核对顶会录用或追踪最新进展，优先按检索来源拆。
  - 如果已有初步证据但某一块薄弱，优先按 gap 拆，只补证据缺口。
  - 每个子 agent 的文件名必须体现拆分依据，例如 `A-LLM工具调用论文.md`、`B-value-of-information经典理论.md`、`C-失败边界与反例.md`。
- 输出：每个 retrieval 子 agent 只返回本领域的来源列表、关键证据、初步 takeaway、证据不足点。

第 1.5 轮：主 agent Triangulation
- 目标：合并第 1 轮结果，判断哪些结论被多个独立来源支持，哪些只是单来源或类比推断。
- 动作：
  - 去重来源；
  - 合并相同证据；
  - 记录冲突结论；
  - 标出证据缺口；
  - 形成待补证 gap list。

第 2 轮：Gap-Fill
- 目标：只针对第 1.5 轮发现的证据缺口补查，不重复覆盖已有结论。
- 启动方式：主 agent 根据 gap list 创建一个或多个 gap-fill 任务。
- 输出：每个 gap-fill 子 agent 必须说明 gap 是否被填补；如果无法填补，要明确写"未找到直接证据"，不能用弱相关材料硬补。

第 3 轮：Verification
- 目标：检查关键断言是否真的被引用来源支持。
- 启动方式：主 agent 抽取最重要的 5-10 条断言，派 verification 子 agent 逐条核查。
- 输出：对每条断言给出 SUPPORTED / PARTIAL / UNSUPPORTED，并说明是否需要改写、降级为综合推断或删除。

第 N 轮：Follow-up Deepening
- 触发：任一轮子 agent 返回 `followup_candidates` 后，由主 agent triage 决定是否开启。
- 目标：只研究被提升的 high-impact 候选问题，例如某个核心单元的跨领域依据、某个断言的实验支持、某个模块的失败边界。
- 启动方式：主 agent 创建或更新对应问题目录下的 `research.md`，指定下一轮子 agent 的任务、范围和输出文件。
- 输出：当前问题的补充证据、边界判断、需要降级的结论，以及新的 `followup_candidates`。新的候选问题仍然只能返回给主 agent，不允许子 agent 继续派生。
```

多轮子 agent 查证时，主 agent 必须为每个子 agent 指定清楚的轮次、角色、查证范围、排除范围和落盘目录。目录放在同一个问题研究目录下，例如：

```text
tmp/01-单agent自我合理化问题研究/
  research.md
  rounds/
    round1-retrieval/
      A-机器学习与LLM论文.md
      B-认知科学与心理学.md
      C-哲学与决策理论.md
      D-经典理论与综述.md
    round2-gapfill/
      gap-01-失败边界补证.md
      gap-02-直接实验支持补证.md
    round3-verification/
      verification-report.md
```

每个子 agent 只能回答自己负责的证据链，不要替其他子 agent 综合结论，也不要跳到下一轮任务。如果发现范围外但重要的问题，只能写成 `followup_candidates`。主 agent 在第 6 节统一读取这些结果，合并重复证据，处理相互冲突的结论，并标明哪些结论是直接支持、哪些只是综合推断。

子 agent 的目标：

> 独立查证，不接着当前笔记继续补全文本。

## 6. 主 agent 综合子 agent 结果

子 agent 返回调研结果后，再让主 agent 做二次综合。

执行要求：

1. 逐个读取 `tmp/*问题研究/` 下的 `research.md` 和 `rounds/`；
2. 先合并 `rounds/round1-retrieval/`：去重来源，按领域整理证据，记录初步共识和冲突；
3. 如果输入是论文、综述或文章，先读取 Round 1 中的"原文结构理解"子任务，并把文章主线、章节 / taxonomy、作者断言、证据组织方式和原文边界作为后续综合的骨架；如果输入是系统或技术方案，先读取"对象结构理解"子任务，并把架构、模块关系、核心设计目标和证据组织方式作为骨架。外部 retrieval 只能补充或校验结构理解，不能替代"对象本身是什么"；
4. 收集所有子 agent 输出中的 `Follow-up Candidates / 后续候选问题`，追加或更新到 `tmp/followup_candidates.md`；
5. 对候选问题做 triage：合并重复候选，标记来源，判断优先级，评估它是否影响最终对内容、核心单元、机制、证据强度或迁移价值的判断；
6. 只把 high-impact 候选提升为下一轮任务；提升时创建或更新对应 `research.md`，记录 `parent_issue`、`derived_from`、`source_subagent_output`、`promotion_reason` 和输出目录；
7. 对 medium / low 候选标记为 deferred / dropped / resolved，不要为了完整性继续派发；
8. 执行 pre-final gate：
   - 如果用户要求"每个创新点 / 每个机制 / 每个 claim / 某一个指定点"的深入分析，必须确认这些目标单元都已出现在 `tmp/followup_candidates.md` 或 `research.md` 的 candidate unit 状态表中；
   - 每个目标单元必须有 promoted / deferred / dropped / resolved 状态和理由；
   - 如果存在 promoted 但未完成的下一轮任务，不允许写最终笔记；
   - 如果 Round 1 子任务输出文件已存在，但 `research.md` 仍为 pending，必须先更新状态；
   - 如果没有创建 `tmp/followup_candidates.md`，必须说明没有候选问题；否则不允许直接 final。
   - **轮次转换门合规检查**：对每一个已执行的轮次（Round 1、Round 2、Verification 等），确认 `research.md` 中存在该轮次的完整子任务清单（编号、角色、方向、状态、输出路径）。如果某个轮次的子任务已执行但 research.md 中没有对应条目，说明轮次转换门被跳过，必须先补录子任务清单和状态再进入 Final。
   
   pre-final gate 结构检查项（追加，根据任务类型触发）：
   - 如果任务类型为"单对象深度分析"或更高深度：
     - 检查 Round 2 是否已按核心单元逐一执行（每个核心单元有独立的 research 目录或在 Round 2 输出中有独立分析段落，包含"痛点→修改点→机制→证据→边界"完整分析链）。
     - 检查跨领域概念映射表是否已在最终笔记中填充（至少 2 个领域覆盖）。
     - 如果以上任一项未完成，禁止进入 Final。
   - 如果任务类型为"多对象对比分析"：
     - 额外检查 Verification 轮是否已执行（至少 5 条关键断言的引用核查，每条标记 SUPPORTED / PARTIAL / UNSUPPORTED）。
     - 检查每个对象的核心单元是否都有独立的"痛点→修改点→机制→证据→边界"分析链。
     - 如果以上任一项未完成，禁止进入 Final。
   - 执行清单逐项核对：
     - 读取 `research_state.md` 顶部的执行清单，逐项确认所有必选项已打勾。
     - 任何未打勾的必选项必须在进入 Final 之前补完，或获得用户明确同意后标记为 deferred。
9. 再检查是否需要 `rounds/round2-gapfill/`：如果某个关键结论只有单来源、缺少反例、缺少最新论文、缺少跨学科支持，必须标成 gap，并派 gap-fill 子 agent 补查；
10. 合并 `rounds/round2-gapfill/`：只吸收真正补上 gap 的证据；无法补上的地方保留为不确定项；
11. 抽取最重要的 5-10 条断言，派 `rounds/round3-verification/` 检查引用是否真的支持该断言；
12. 根据 verification 结果处理断言：
   - SUPPORTED：可以保留为文献直接支持；
   - PARTIAL：必须改写为更弱、更准确的表述；
   - UNSUPPORTED：删除、降级为综合推断，或继续标为未证实；
13. 更新 `research.md` 中的子任务状态、follow-up candidate 状态、gap list 和 verification 状态；不要把子 agent 原始结果复制进 `research.md`；
14. 区分"文献直接支持"和"综合推断"；
15. 改写成研究笔记风格；
16. 根据问题需求压缩：优先写成"对象—问题—依据—路径 / 机制 / 映射—边界—启发"；创新类可加入旧方法缺陷和设计哲学，判定类可加入判定标准和结论强度，综述类可加入分类框架和覆盖范围；
17. 使用更学术、严谨的中文表达；
18. 保留关键术语，去掉泛泛总结；
19. 把可复用内容合并进最终笔记结构；
20. 保留 `tmp/*问题研究/` 作为可追溯的子问题研究档案，不要用最终笔记覆盖或替代它们。

目的：

> 把调研材料转化成自己的知识结构。

## 7. 最终笔记格式

最终笔记只负责综合和压缩，不替代 `tmp/*问题研究/` 下的子问题调研文件；每个问题的检索记录、证据和完整调研结论仍应保留在对应目录中。

最终组织方式由主 agent 根据用户问题、材料结构和子 agent 的实际拆分决定。不要为了套模板强行写多个问题；如果只有一个综合问题，就只写一个分析单元。如果子 agent 是按章节、模块、证据来源、学科或机制链拆分的，最终笔记也可以按这些单元组织，而不是必须写成"问题 1 / 问题 2"。

输出格式：
```md
## 【对象名称】

本质：
- 一句话本质

- **通用分析骨架**
	- **对象与边界**：
		- 它真正研究 / 解释 / 判定 / 映射的对象是什么。
		- 它不研究什么；哪些内容只是背景或辅助材料。

	- **核心问题 / 主线**：
		- 它要回答什么问题。
		- 如果是综述：它如何组织领域和 taxonomy。
		- 如果是系统 / 方法：它要解决什么缺陷或瓶颈。

	- **依据与可信度**：
		- 它凭什么成立：原文、实验、对照测试、性能评估、理论证明、综述、经典理论、复现结果或外部工具。
		- 区分文献直接支持、综合推断和仍不确定的部分。

	- **路径 / 机制 / 组织方式**：
		- 如果是机制解释：因果链是什么。
		- 如果是综述或文章理解：章节、观点、分类和证据如何组织。
		- 如果是系统分析：模块如何连接，信息如何流动。
		- 如果是判定或概念映射：标准或映射关系如何作用到证据上。
```
